這是一個結構明確且必要的擴展。總經分析（Macroeconomic Analysis）的加入，能將單純的技術面與基本面評估提升至更宏觀的投資週期判斷層級，極大地增強了系統的深度和可信度。

## 📈 總經分析子 Agent 架構規劃 (MacroAgent)

### I. 系統流程圖與職責劃分

新的 `MacroAgent` 將作為一個獨立的「觀點生成層」（View Generation Layer），其輸出是高度結構化的、可供後續整合的文本摘要，而非原始數據。

**【新增工作流】**
1. **輸入 (Input):** 股票代碼 $\rightarrow$ 所屬產業別 $\rightarrow$ 市場類型 (TW/US) $\rightarrow$ 分析時間週期 (e.g., Q3 2024)。
2. **MacroAgent 執行:**
    *   A. 數據擷取：獲取全球關鍵指標（如 CPI, PMI, Fed Rate, 貿易指數等）。
    *   B. 指標分析：判斷當前宏觀環境處於哪個週期階段 (e.g., 通縮、滯脹、復甦)。
    *   C. 產業/市場映射：根據 A 和 B 的結果，對所屬的產業別和市場類型進行風險評級與趨勢預測。
3. **輸出 (Output):** 一份結構化的 JSON 或 Markdown 格式的「宏觀總結報告」，傳遞給綜合分析 Agent。

### II. 模組化設計細節與數據流管理

#### A. `MacroDataFetcher` 模組（資料層）
此模組負責從多源頭獲取全球經濟指標。

*   **核心功能：** 定時同步 CPI、PMI (購買經理人指數)、利率曲線、主要國家 GDP 增長率等數據。
*   **技術考量與防禦要點執行：**
    1.  **【Unicode 防禦】:** 若從本地 CSV 或資料庫讀取歷史指標數據，必須在所有 `open()` 函數中強制加入編碼處理。例如：`pd.read_csv(file_path, encoding='utf-8', errors='ignore')`。
    2.  **【JSON 序列化保護】:** 當將多個時間序列的宏觀數據（包含日期和浮點數）組合成一個待傳輸的 Python Dictionary 時，必須預先執行格式轉換：
        *   `datetime.date/datetime.datetime` $\rightarrow$ `str` (ISO Format)。
        *   `decimal.Decimal` $\rightarrow$ `float` 或 `str`。

#### B. `ViewGeneratorAgent` 模組（推理層）
這是核心的 LLM 推理環節，負責將原始數據轉化為可讀的投資觀點。

*   **輸入：** 結構化的宏觀指標數據 (JSON)。
*   **處理邏輯：** 採用 RAG 或 Prompt Engineering 的方式，設計一套決策樹或規則集：
    *   *IF* CPI 上升 $\text{AND}$ PMI 下降 $\rightarrow$ **可能進入滯脹環境。**
    *   *IF* 市場為美股 $\text{AND}$ 環境為通縮 $\rightarrow$ 建議關注價值型、抗衰退產業 (Defensive Sector)。
*   **輸出格式：** 必須是高度結構化的 JSON，包含以下欄位：

```json
{
  "macro_summary": "總體經濟環境判斷",
  "global_outlook": {
    "cycle_stage": "e.g., Early Recovery / Stagflation",
    "risk_level": "Medium-High",
    "key_drivers": ["Inflationary pressure from energy.", "Rate hike cycle nearing end."]
  },
  "market_specific_view": {
    "market_type": "US Stock Market",
    "sector_recommendation": "Technology (AI Infrastructure)",
    "rationale": "受惠於全球數位轉型，且利率環境趨穩利於高成長科技股估值。"
  },
  "actionable_advice": "建議在短期內關注具有強大現金流和定價能力的領導品牌。",
}
```

### III. 整合與防禦要點總檢核 (Critical Review)

為確保新功能穩定運行，必須嚴格遵守以下專案歷史踩坑防禦要點：

| 防禦要點 | 發生情境 | 執行機制要求 |
| :--- | :--- | :--- |
| **【Unicode 防禦】** | `MacroDataFetcher` 從本地讀取全球指標 CSV 或資料庫。 | 所有檔案 I/O 必須使用 `errors='ignore'` 或 `errors='replace'`，避免編碼衝突導致程式崩潰。 |
| **【JSON 序列化保護】** | 將 MacroAgent 的分析結果（包含日期、浮點數）傳遞給後續的 API 呼叫或資料庫儲存時。 | 必須在 Python 層面強制將 `datetime` 和 `Decimal` 轉換為標準的 `str` 或 `float`，防止序列化失敗。 |
| **【Gemini API 400 兼容】** | 當綜合分析 Agent 接收到 MacroAgent 的觀點後，若需要進行二次推理（例如：將宏觀觀點與技術圖表結合），且使用 Function Calling 機制。 | 第二輪 `functionResponse` 必須完整包含第一輪所有傳入的 `parts` (包括原始輸入和任何思維簽名 $\text{thoughtSignature}$)，否則會觸發 HTTP 400 錯誤。 |
| **【零破壞原則】** | 新增的總經分析面板或報告。 | 此模組必須設計為一個獨立的、可選啟動的「資訊懸浮區」（Information Overlay Panel）。其輸出應僅作為輔助參考，絕不能修改或干擾現有的 ApexCharts K 線圖繪製邏輯或 `blended valuation` 的計算核心函數。 |

### 總結建議

請將 MacroAgent 設計為一個**異步（Asynchronous）**的服務呼叫。當用戶請求分析時，系統應先啟動技術面和基本面分析，同時在背景啟動 MacroAgent。最終結果由主控 Agent 負責協調並以統一、分層次的介面呈現給使用者。
