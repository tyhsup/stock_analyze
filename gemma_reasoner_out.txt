這是一個典型的金融圖表視覺化問題，當一個圖表需要同時呈現兩個或多個具有不同單位、不同量級的指標（例如：價格指數與交易量/CPI），若不進行軸向同步處理，將會導致視覺失真和誤判。

由於您提到的是「Y軸雙端同步」，我們必須從**資料層 (Data Layer)** 和 **呈現層 (Presentation Layer)** 兩個角度進行結構性修正。這不是單純的程式碼調整，而是需要確保數據傳輸與圖表配置邏輯都符合金融分析的最佳實踐。

---

### 📊 架構分析與修正建議

#### I. 資料處理層 (Data Preparation / Backend)
**核心問題：** 圖表呈現的兩個指標（Metric A 和 Metric B）在資料庫查詢時，其量級差異過大或單位不一致，導致前端接收到的原始數據無法直接用於繪製。

**修正要點：** 在將數據從 MySQL 提取並準備傳輸給 API 之前，必須執行標準化和正規化處理。

1.  **單一時間軸對齊 (Time Alignment):** 確認所有指標的資料點都以統一的時間戳記（Timestamp）作為鍵值進行對齊。
2.  **單位轉換與歸一化 (Normalization/Scaling):** 如果兩個指標量級差異極大（例如：價格在 100-500，交易量在 $10^6$ 到 $10^9$），建議考慮將其中一個或兩個指標進行 **Z-Score 標準化** 或 **對數轉換 (Log Transformation)**，以確保它們的視覺權重不會被單純的絕對值主導。
3.  **【JSON 序列化保護】執行：** 在此階段必須嚴格遵守 JSON 序列化的規則。所有從 MySQL 查詢返回給 API 的 `datetime`、`decimal.Decimal` 等類型，都必須在傳輸前轉換為標準的 **ISO 8601 字串格式 (String)** 或 **浮點數 (Float)**。

#### II. 前端呈現層 (Visualization / Frontend)
**核心問題：** 如何讓圖表庫（如 ApexCharts）知道這兩個 Y 軸是相關聯的，並且在極值發生時保持視覺上的連貫性。

**修正要點：** 利用圖表庫提供的多 Y 軸配置功能，並手動控制其範圍 (Range)。

1.  **Y-Axis 配置 (Multi-Axis Setup):**
    *   將兩個指標分別定義為 `yaxis` 陣列中的獨立組件。
    *   如果它們的單位和量級差異極大，應考慮使用 **次座標軸 (Secondary Y-axis)** 的方式來繪製，而不是讓它們共用一個主軸。

2.  **同步範圍邏輯 (Range Synchronization Logic):**
    *   單純設定 `min` 和 `max` 是不夠的。您需要實作一個計算機制：當數據集發生變化時（例如滾動視窗或時間範圍改變），前端必須重新計算兩個指標在當前時間區間內的 **極小值 (Min)** 與 **極大值 (Max)**。
    *   **關鍵步驟：** 確保這兩個 `min` 和 `max` 值被傳遞給圖表庫的配置，強制設定兩個 Y 軸的範圍 (`yaxis[0].min`, `yaxis[1].min`, etc.) 為相同的計算結果。

3.  **【零破壞原則】考量：**
    *   所有新的同步邏輯和輔助介面（如控制軸向比例的滑桿）必須設計為獨立於現有的 K 線圖與 Blended Valuation 計算區塊，確保使用者操作不會意外干擾核心分析功能。

### 🛠️ 實作流程總結 (Checklist)

| 層級 | 任務描述 | 技術重點 / 需注意的防禦要點 |
| :--- | :--- | :--- |
| **資料層** | 數據清洗與標準化 | 1. 統一時間戳記格式。2. 對量級差異大的指標進行正規化處理 (Log/Z-Score)。3. **【JSON 序列化保護】**：確保所有日期和 Decimal 類型已轉為 String/Float。 |
| **API 層** | 數據傳輸介面設計 | 建立一個專門的 `get_synchronized_data(start_date, end_date)` 端點，該端點負責執行上述所有清洗與計算邏輯。 |
| **前端層** | 圖表渲染與互動邏輯 | 1. 使用圖表庫的 Multi-Axis 功能。2. 在時間範圍改變時（`onZoom`/`onChange` 事件），觸發一個重新計算 `min/max` 的函式。3. 將計算出的同步範圍強制設定給所有相關聯的 Y 軸組件。 |

**總結：** 「Y軸雙端同步」的核心不在於繪圖，而在於 **數據傳輸前的預處理 (Data Pre-processing)** 和 **前端事件觸發時的動態範圍重設 (Dynamic Range Reset)**。請務必在資料層級執行嚴格的類型轉換和正規化，以避免後續的視覺錯誤。
